# Lanternhouse Agent Guide

Lanternhouse is a Godot 4 pixel RPG. Keep changes small, playable, and easy to
merge across worktrees.

## Repo Habits

- Use `rg` first for code and file searches.
- Read the surrounding code before editing; prefer local patterns over new
  abstractions.
- Use `apply_patch` for manual edits.
- Do not revert unrelated worktree changes. Treat unexpected changes as user
  work unless clearly generated by the tool you just ran.
- Keep commits scoped to intentional source, docs, scene, or asset changes.
- After Godot runs, check for generated `.import`, `.uid`, and incidental
  `project.godot` churn before staging.

## Godot Input Pattern

- World/gameplay scripts should generally open menus from `_unhandled_input()`.
- Active full-screen UI overlays should own close/navigation keys in `_input()`.
- When a UI overlay consumes a key, call `get_viewport().set_input_as_handled()`.
- Do not trust a manual `_unhandled_input()` call as proof that a live UI overlay
  will receive the same key. Model Godot's input phase in smoke tests.

## Verification

Run the smallest relevant checks for the change. Useful commands:

```powershell
godot --headless --path . --quit
godot --headless --path . scenes\dev\smoke_overworld_command_menu.tscn
godot --headless --path . scenes\dev\smoke_first_route_guidance.tscn
godot --headless --path . scenes\dev\smoke_dead_wick.tscn
```

For input fixes, add or update a smoke test that exercises the same input phase
used by the live game.

## PixelLab Art

- PixelLab art is the preferred source for production pixel art.
- Use `assets/sprites/characters/cat/` as the current quality benchmark.
- Preserve useful PixelLab folder structure such as `rotations/` and
  `walk/<direction>/`.
- Prefer imported texture resources over runtime `Image.load()` paths where
  export behavior matters.
- When generated art dimensions differ from older placeholder conventions, adapt
  in code rather than destructively resizing source art.

## Worktrees

- Keep `I:\code\lanternhouse` as the `main` integration worktree.
- Use one task or lane per worktree, then merge back to `main` deliberately.
- See `docs/worktree-workflow.md` for creation, update, merge, and cleanup
  commands.
